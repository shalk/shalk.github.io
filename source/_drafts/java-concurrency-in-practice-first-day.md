## 第一章

从为什么需要并发说起，以前是单处理器运行一个程序，这十分浪费资源，于是产生了操作系统可以运行多个程序，每个程序是一个进程，操作系统可以对进程进行调度，有信号量，共享内存等机制，用于提高资源利用率，公平性等，而这些因素也促使了线程的出现，一个进程可以有多个线程，线程相当于轻量级的进程。进程有很多的优势，提高资源吞吐率，简化编程建模，异步事件处理简单，GUI响应简单。同时进程也会产生一些问题，安全性（多个线程同时访问相同的状态），活跃性问题（例如 死锁，活锁，饥饿），性能问题（线程切换，锁的分配释放）。在java的世界，许多库和框架都使用了线程，例如jvm，servlet，rmi，swing，还有很多rabbitmq-client，spring-boot都使用了线程。

简单来说，线程好处多，线程也危险，但是不用怕，很多人都在用，下面有一些解决危险的办法。

## 第二章

线程安全的基础理论。

线程安全的程序，本质上是在多线程运行环境下，要对状态的访问进行管理，尤其是共享和可变的。

如果多个线程访问并修改，对象的可变状态，那线程可能出现不正确的行为，一般认为这时是不安全的。

可以通过许多同步机制（不仅限于synchronized）来保证多个线程对同一个可变状态进行访问。

```
如果多个线程访问同一个可变的状态变量没有合适的同步，那么程序就会出现错误。
有三个办法解决：
- 不再线程间共享
- 将状态变量改为不可变
- 在访问状态变量时进行同步。
```

后面所有的解决线程安全的办法，都是基于三个简单的办法。

实际上并不那么容易，一般实际套路都是从设计线程安全类开始，封装许多线程安全的类，（虽然线程安全的程序并不全由线程安全的类组成，而且线程安全的类，组成的程序也不完全是线程安全的，注意，这里说的是调用端使用了一些线程安全的类），但按照这个方式，可以极大降低并发编程的复杂度，而且有很多类库，框架都是这样实践的。

当我们使用调用线程安全类，编写自己的多线程程序时，可以参考接下来的套路和理论进行设计，避免线程安全的问题。

什么是线程安全的类：

```
定义：但多个线程访问某个类时，不管运行环境采用何种调度方式或者写线程如何交替执行，并且在主调代码中不需要任何额外的同步或者协同，这个类都能表现出正确的行为，那么这个类就是线程安全的。

正确的行为：某类的行为与其规范安全一直。 

规范是：不变性（约束对象的状态）和后验条件。
```

惯例： 无状态的类是线程安全的。

race condition: 竞态条件是一种非线程安全的情况，多个线程共享资源，由于不恰当的执行顺序引起不正确的结果。

原子操作：

```
假定有两个操作A和B，如果从执行A的线程来看，当另一个线程执行B时，要么将B全部执行完，要么完全不执行B，那么A和B对彼此来说是原子的。原子操作是指，对访问同一个状态的所有操作（包括该操作本身）来说，这个操作是一个原子方式执行的操作。
```



常见的race condition 有，读取-修改-写入，先检查后执行（CTA）

比如解决读取-修改-写入（i++)

把这个复合操作，使用同步机制，把复合操作变成原子操作。

有两个办法，

第一，使用线程安全的类（如果类只有一个状态，那状态直接用线程安全的类，那这个类也是线程安全的），AtomicInteger。

需要注意的是：如果有两个状态，而且两个状态之间有不变性规范，单纯用线程安全类是不行的，加锁的方式可能更合适。

另一种办法，就是加锁进行同步。

内置锁：synchronize 加到方法体或者  加到this上，每个对象都有一个内置锁，这种比较方便。实际上，用任何别的对象当锁也可以。加锁的地方是同步代码块，一般只有一个对象可以进入同步代码块。

为了解决，获得锁之后又调用其他加锁的方法的死锁问题，内置锁是重入的，也就是获得了锁的线程可以继续进入同一个锁的同步代码块。

加锁要注意几点：

1. 同一个状态访问如果要加锁，需要加同一个锁，而且所有访问的地方都要加。

2. 不变性规范的多个状态加锁，也要加同一个锁。

3. 锁和状态没有内在联系，需要自己设定同步规范。

4. 某个类的所有把所有状态都封装到内部，所有方法都添加synchronized到方法原型上，是一种常见办法，这样类是线程安全的。

5. synchronzed不可以滥用，虽然简单，但是可能引起性能问题。最好是要进行充分考虑和优化，在需要添加的地方加锁，做到恰到好处。

## 第三章，对象的共享

前面一个章节有介绍，多线程在共享资源时，因为不正确的时序，发生不正确的行为，叫竞态条件。

并且最开始说道，一种不安全的情况，就是多个线程访问修改对象的可变状态，发生不正确的行为。

是的，但讨论线程安全时，要考虑到一个重要因素的是资源共享，或者说对象共享。

### 可见性

在讨论对象共享时，有一个重要理论，就是可见性(visibility)，由于java的内存模型原因，必须要考虑可见性。

多个线程在访问同一个对象，都会产生该对象的缓存，这样导致一个线程在修改数据之后，另外一个线程可能还会访问到失效数据，使得结果不符合预期。

同时由于jvm会指令重排，更会引起不正确的情况出现。（本书不讨论指令重排）

解决可见性的有两个办法。

- volatile
- 加锁（synchronize 内置锁）

需要注意两点：

1. 虽然当数据可以修改，会产生失效数据的时候才需要解决可见性问题，不可变对象貌似不需要解决不可见问题，通常我们使用reference type,refrence 可能会改变，因此需要volatile 修饰，例如String 对象的引用。
2. 但且仅当满足以下所有条件的时，才使用volatile
   - 对变量的操作不依赖当前值，或者 只有一个线程更新变量
   - 不与其他变量纳入不变性条件
   - 不需要加锁

### 不变性

不变性是一个非常重要的性质，但一个对象在创建之后，状态不会发生改变，称为为不可变对象。

不可变对象的状态包括所有的域，以及域中对象的状态。

不可变对象需要满足以下条件：

- 所有的域都是final关键字
- 对象创建以后，就是不可修改的
- 对象是正确创建的（构造过程中没有this逸出）

final的作用是：变量赋值后，不可以改变。（通常是reference type 和 primitive type 不可变）
